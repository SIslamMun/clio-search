# Plan 03-03: Federated Multi-Namespace Search + Multi-Hop Agentic Retrieval

## Context
This plan covers the final two design components: federated multi-namespace search (the connector architecture that enables unified search across heterogeneous HPC storage) and multi-hop agentic retrieval (the iterative LLM-driven search loop). These are contribution items 3 and 5 from the introduction.

## Tasks

<task type="auto" mode="co-author">
  <name>Connector protocol and capability negotiation</name>
  <action>Write ~125 words describing the connector protocol. Two interfaces: NamespaceConnector (connect, teardown, index, descriptor, build_citation) and RetrievalCapabilities (search_lexical, search_vector, search_graph, search_scientific). Each connector declares its capabilities at registration. Capability matrix: FilesystemConnector (lexical, vector, scientific), S3ObjectStoreConnector (lexical, vector, scientific), QdrantConnector (vector only), Neo4jConnector (graph only), HDF5Connector (metadata, scientific), NetCDFConnector (metadata, scientific). The coordinator queries each namespace for its capabilities and routes only supported retrieval branches to each backend -- a filesystem gets BM25+vector+scientific queries while Qdrant gets only vector queries.</action>
  <verify>Connector types and capabilities match connectors/ directory implementations. Protocol interfaces match the codebase.</verify>
</task>

<task type="auto" mode="co-author">
  <name>Multi-namespace query flow</name>
  <action>Write ~75 words describing the query execution flow across namespaces. Query arrives -> coordinator iterates over registered namespaces -> for each namespace, executes a single-namespace retrieval pipeline (only branches that namespace supports) -> collects all citations from all namespaces -> global sort by composite score -> deduplicate across namespaces -> select top-K -> return with per-namespace provenance. Emphasize: science-aware operators apply identically across all backends because canonicalization happens at index time.</action>
  <verify>Flow matches retrieval/coordinator.py query execution logic.</verify>
</task>

<task type="auto" mode="co-author">
  <name>Incremental indexing</name>
  <action>Write ~50 words on incremental indexing. SHA256 content hash + mtime change detection per file. On re-index: skip unchanged files, re-index changed files, remove entries for deleted files. Stored in DuckDB file_index table. Enables efficient indexing of evolving scientific corpora without full re-indexing.</action>
  <verify>Matches storage/duckdb_store.py file_index table and indexing logic.</verify>
</task>

<task type="auto" mode="co-author">
  <name>Multi-hop agentic retrieval: iterative loop</name>
  <action>Write ~100 words describing the multi-hop agentic retrieval loop. Process: (1) Initial query -> execute full hybrid pipeline -> retrieve top-K results. (2) LLM evaluates relevance of results against the original query intent. (3) LLM selects a rewriting strategy and generates a refined query. (4) Re-execute full pipeline with refined query. (5) Merge new results with accumulated results. (6) Repeat until convergence (results stabilize across hops) or max_hops reached (configurable, default=3). Each hop is a complete pipeline execution -- the agentic loop wraps the single-pass pipeline.</action>
  <verify>Loop structure matches implementation. Configurable max_hops is accurate. Convergence detection is implemented.</verify>
</task>

<task type="auto" mode="co-author">
  <name>Query rewriting strategies and LLM integration</name>
  <action>Write ~100 words describing the query rewriting strategies and LLM integration. Four strategies: (1) Expand: add related terms, synonyms, or unit variants discovered in initial results. (2) Narrow: focus on a specific subquery when initial results are too broad. (3) Pivot: shift to a related measurement or formula when direct match fails. (4) Done: results are sufficient, stop iterating. LLM integration via Anthropic API (Claude) for strategy selection and query generation. Fallback mode: offline SI unit variant expansion when LLM is unavailable -- ensures the agentic loop functions without external API dependencies.</action>
  <verify>Four strategies match implementation: expand/narrow/pivot/done. Fallback mode is implemented. Anthropic API integration is real.</verify>
</task>
