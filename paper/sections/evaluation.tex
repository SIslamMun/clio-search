\section{Evaluation}
\label{sec:evaluation}

We evaluate CLIO along three axes: (1)~agentic efficiency---does CLIO
reduce LLM token consumption and latency compared to raw catalog access?
(2)~science-aware accuracy---do dimensional conversion and formula
normalization improve retrieval precision where text-based methods fail?
(3)~storage integration and scaling---can CLIO serve as the search layer
for IOWarp's tiered blob storage and scale across distributed HPC nodes?

\subsection{Experimental Setup}
\label{sec:setup}

\textbf{Platforms.} Single-node experiments run on Linux 6.17 with
Python 3.11, DuckDB 1.1, and all-MiniLM-L6-v2 embeddings (384-dim).
Distributed experiments run on NCSA DeltaAI (NVIDIA GH200 Grace
Hopper nodes) with a 2.5M arXiv abstract corpus. IOWarp experiments
run inside Docker with the \texttt{iowarp/deps-cpu} image and
Chimaera runtime v1.0.3.

\textbf{Controlled benchmark.} 210 documents across seven scientific
domains (Table~\ref{tab:dataset}) with deliberate unit heterogeneity:
the same physical quantities are expressed in different SI prefixes.
80~queries with automatically generated ground truth. We additionally
validate on 288 real NOAA observatory documents~\cite{ghcn2012}.

\textbf{Baselines.} BM25-only (DuckDB full-text search); Dense-only
(all-MiniLM-L6-v2, cosine similarity); Hybrid BM25+Dense (linear
fusion); String normalization~\cite{numbersmatter2024}; and CLIO's
full pipeline.

\textbf{Metrics.} P@$K$, R@$K$, MRR at $K{=}5$ for retrieval accuracy.
Total tokens (including sub-agent tokens) and wall time for agentic
efficiency.

\subsection{Agentic Efficiency}
\label{sec:agentic-efficiency}

We run 10~scientific discovery queries through three paths using
\texttt{claude-sonnet-4-20250514} and the National Data Platform (NDP)
catalog: (1)~a \textbf{Claude Native Agent} with no domain tools (uses
web search and sub-agents), (2)~an agent with \textbf{NDP-MCP} catalog
tools, and (3)~an agent with a single \textbf{CLIO} tool that indexes
341~NDP datasets into DuckDB and returns ranked citations.

CLIO reduces tokens by \textbf{52.08\%} and wall time by
\textbf{91.27\%} (Table~\ref{tab:l1-results}). The native agent and
NDP-MCP consume $\sim$989K tokens through different mechanisms: the
native agent spawns 9~sub-agents (563K main + 426K sub-agent tokens);
NDP-MCP injects raw catalog JSON (989K main tokens). CLIO avoids both
by indexing catalog content locally and returning $\sim$2K tokens of
ranked citations per query. NDP-MCP reduces wall time by 66\% over the
native agent by querying the API directly, but achieves no token
savings. CLIO reduces time by a further 75\% over NDP-MCP
(Fig.~\ref{fig:l1-three-path}).

\subsection{Science-Aware Retrieval Accuracy}
\label{sec:accuracy}

\textbf{Cross-unit retrieval.}
On 40~cross-unit queries (Table~\ref{tab:crossunit}), text baselines
cannot match across SI prefixes: a query for ``200~kPa'' never
lexically matches ``200000~Pa.'' BM25 achieves P@5 of 0.40; our
pipeline canonicalizes to SI base units at index time and achieves
\textbf{P@5 of 0.99} (+0.59 absolute). On same-unit controls, all
methods score within 0.05 P@5, confirming gains are specific to
cross-unit matching.

\textbf{Formula matching.} On 10~formula queries, BM25 achieves R@5 of
0.48 via loose text matching; CLIO achieves R@5 of 0.10 via strict
symbolic equivalence. CLIO underperforms BM25 on this task because its
formula operator requires exact structural equivalence after
canonicalization, while BM25 matches formula-related tokens loosely.
This is a deliberate precision-over-recall trade-off: CLIO produces
fewer but more reliable matches. Improving formula recall without
sacrificing precision is future work.

\textbf{Ablation.} Adding science operators (B$\to$C in
Table~\ref{tab:ablation}) produces the dominant gain: P@5 jumps from
0.36 to 0.99 (+0.63). The full pipeline (D) adds no further gain on
this benchmark, confirming that cross-unit matching is the bottleneck.
A no-LLM fallback achieves 96\% of LLM-assisted P@5, confirming that
the operators---not the language model---drive gains.

\textbf{Metadata-adaptive strategy.} CLIO correctly classifies three
corpus variants (rich 87.5\% density $\to$ scientific branch, sparse
10.3\% $\to$ lexical+vector, no-metadata 4.4\% $\to$ lexical only)
and activates unit conversion only when extractable measurements exist.
Federated search across filesystem, S3, and HDF5 backends doubles
coverage from 24.4\% to 47.2\%, with science operators composing
without degradation via capability negotiation.

\subsection{Storage Integration and Scaling}
\label{sec:storage-scaling}

\textbf{IOWarp blob storage.}
At IOWarp's target scale of 10~billion blobs, the native BlobQuery API
returns all blobs matching a tag regex with no content filtering. CLIO
addresses this by indexing scientific metadata from CTE blobs at ingest
time. We run the Chimaera runtime (v1.0.3) and store synthetic blobs
across four CTE tags (temperature in \textdegree C, pressure in kPa,
wind in m/s). Four cross-unit queries use mismatched units
(\textdegree F, psi, km/h).

top-50 unit-converted matches; BlobQuery returns all blobs unfiltered.}
\label{tab:iowarp}
\small
\begin{tabular}{rrrrrr}

CLIO finds cross-unit matches at every scale while BlobQuery cannot
filter by content (Table~\ref{tab:iowarp}). The inspection rate falls
from 5.0\% at 1K to \textbf{0.5\% at 10K} ($50\times$ reduction over
BlobQuery). Corpus profiling completes in 14--22\,ms regardless of
scale.

\textbf{Cross-unit correctness at distributed scale.}
To verify that dimensional conversion remains correct when the corpus
is distributed, we issue cross-unit probe queries against the 70K-document
distributed corpus (4~shards on DeltaAI GH200). Table~\ref{tab:d4-crossunit}
shows that querying in different units yields \emph{identical hit counts}:
100~kPa, 100{,}000~Pa, 1~bar, and 0.987~atm all return exactly 12~hits;
30\textdegree C, 86\textdegree F, and 303.15~K all return exactly
20~hits. This confirms that SI canonicalization is consistent across
shards---the coordinator merges results correctly regardless of which
unit the query uses.

\textbf{Numeric query coverage at scale.}
We run 20~scientific numeric queries spanning 16~unit types (GeV, nm,
GHz, kelvin, kPa, tesla, etc.) against the distributed corpus
(Table~\ref{tab:d5-numconq}). All 20~queries return results with
\textbf{100\% hit rate}, p50 latency of 42\,ms, and p99 of 62\,ms.
This demonstrates that CLIO's unit registry generalizes beyond the
controlled benchmark to real scientific literature at scale.

\textbf{Limitations.} The regex extractor misses non-standard formats
(``$2 \times 10^3$~Pa''), achieving 95\% precision / 90\% recall.
Compound units (N/m$^2$, mol/L) are unsupported and fall back to
lexical retrieval. The 46-unit registry (12~SI domains) is extensible
by adding rows without code changes.
