\section{Introduction}

When a scientist searches for data across a large repository, they do not simply type a query and read results. They inspect what each dataset contains, determine whether it has structured metadata or only raw files, decide which search approach will work for each dataset, apply domain knowledge to interpret measurements across different conventions, and refine their search when initial results are insufficient. This process is reasoning, not retrieval. It is how a human expert navigates heterogeneous scientific data. No retrieval system replicates it.

At HPC scale, human reasoning does not scale. A single exascale job on Frontier can produce 80~PB of simulation output~\cite{george2025lustre}. A commonly cited estimate holds that 60 to 80\% of scientific data is never reused, not for lack of access, but for lack of search~\cite{baker2016reproducibility}. The FAIR data principles~\cite{wilkinson2016fair}, now mandated by NSF, NIH, and DOE~\cite{nih_dmsp2023}, require that data be Findable, Accessible, Interoperable, and Reusable, yet automated assessments show no major repository achieves 50\% reusability compliance. The retrieval infrastructure that serves these scientists treats all content as plain text.

AI agents are now entering this space. Coscientist~\cite{coscientist2023} executes lab synthesis autonomously, AI Scientist v2~\cite{aiscientistv2_2025} generates research end-to-end, and autonomous agents at ORNL~\cite{ornl_agents_sc25} orchestrate experiments across national facilities. But ScienceAgentBench~\cite{scienceagentbench2025} reports 32.4\% end-to-end task success, with failures traced to data handling rather than reasoning. A comprehensive survey~\cite{agentic_ai_survey2025} identifies scientific data management as explicitly underexplored. These agents inherit the same retrieval infrastructure that fails human scientists: the retrieval layer underneath them is blind to scientific structure.

Current retrieval systems fall short of scientific data discovery in three fundamental ways.

First, scientific search requires domain reasoning that retrieval systems do not perform, a gap we call \emph{dimensional blindness}. A query for pressures between 190 and 360~kPa must recognize that ``200000~Pa'' and ``0.2~MPa'' are both matches, but this requires arithmetic, not text similarity. Formulas compound the problem: ``$F = ma$,'' ``$F = m \cdot a$,'' and ``$ma = F$'' express the same physical law but produce distinct token sequences that defeat both keyword and embedding retrieval. The Numeracy Gap study~\cite{numeracygap2026} evaluates 13 embedding models and reports 0.54 binary retrieval accuracy on numerical content. String normalization~\cite{numbersmatter2024} cannot bridge SI prefixes. Learned approaches~\cite{cone2026,ncretriever2025} achieve up to 65\% recall improvement but provide no correctness guarantee. The consequences are real: NASA's \$327M Mars Climate Orbiter was lost because one system used pound-seconds while another used newton-seconds~\cite{stephenson1999mco}; the Gimli Glider incident stemmed from metric vs.\ imperial fuel calculations~\cite{ward2021gimli}; and medication errors, including unit conversion mistakes, cause thousands of deaths annually in US healthcare~\cite{iom2006medication}.

Second, scientific datasets exhibit extreme \emph{metadata heterogeneity}, and no search system adapts to this variation. Automated FAIR metadata assessments consistently find that fewer than half of datasets in major repositories carry structured, machine-readable annotations~\cite{baker2016reproducibility}. An HDF5 file with Climate and Forecast convention attributes provides explicit units for every variable; a raw binary dump on a parallel filesystem may offer nothing beyond a filename. A fixed retrieval pipeline cannot handle both: it either wastes computation on datasets with no extractable measurements or misses opportunities when rich metadata is available. A search system that profiles each corpus before querying, determining what metadata exists and which strategies are viable, would address this gap, but no current retrieval system provides this capability.

Third, \emph{storage fragmentation} across heterogeneous backends places scientific data beyond the reach of any single search engine. A single experimental campaign may deposit raw output on a parallel filesystem, preprocessed arrays in HDF5, derived features in a vector database, and catalog entries on an external platform. RAGRoute~\cite{ragroute2025} routes queries to relevant data sources via a learned classifier, but searches only over unstructured text. SMURF~\cite{smurf2025} federates retrieval across scientific repositories by aligning embeddings, but treats data as opaque vectors without inspecting structured metadata. Neither uses metadata to direct search with domain-aware operators. The difficulty is that each backend exposes different metadata: an HDF5 file carries CF-convention unit attributes, a filesystem offers only filenames, and a vector database stores embeddings with no unit semantics. A federated search system must adapt its operators to what each backend can provide. No existing system combines domain-aware operators, corpus-adaptive orchestration, and federated HPC access.

These challenges share a common requirement: the search system must reason about \emph{how} to search before it executes the search. It must observe what each corpus contains, decide which strategies to apply, act with domain-specific operators, and evaluate whether to refine. This observe-decide-act-evaluate cycle is what separates agentic search from conventional retrieval. It is how human scientists search. No retrieval system implements it.

We present CLIO Search, a retrieval system that implements this cycle for scientific data. CLIO profiles each corpus to discover what metadata exists and selects search strategies based on what the corpus can support. It executes science-aware operators, dimensional conversion and formula normalization, alongside standard text and vector search, and federates across heterogeneous HPC storage backends. An agentic refinement loop expands, narrows, or pivots queries based on intermediate results. Domain knowledge enters the pipeline as verifiable computation: the fact that 5~km = 5000~m is an algebraic invariant, guaranteed correct by construction. We evaluate CLIO on controlled benchmarks, real observatory data, and distributed HPC infrastructure at scale. CLIO achieves 0.99~P@5 on cross-unit queries versus 0.40 for the strongest text baseline. On real NOAA observatory data, precision improves by 87\%. At 2.5M~arXiv abstracts across 4~GH200 nodes on NCSA DeltaAI, cross-unit queries return in under 50\,ms.
Our contributions are as follows:

\begin{enumerate}
\item \textbf{Science-aware retrieval operators.} Dimensional conversion and preliminary formula normalization ($F{=}ma$ vs.\ $ma{=}F$) execute as deterministic operators alongside BM25 and dense search. A unit registry converts measurements to SI base units through arithmetic multiplication, guaranteeing correctness by construction. The registry covers 58~units across 12~SI domains\footnote{Length, mass, time, temperature, pressure, velocity, frequency, radioactivity, energy, power, area, acceleration.} and extends with one configuration entry per unit.

\item \textbf{Corpus-adaptive search orchestration.} Before querying, CLIO inspects what metadata each corpus provides and selects search strategies accordingly. Corpora with structured measurements activate science-aware operators; corpora with no extractable metadata fall back to text retrieval.

\item \textbf{Federated scientific search.} A connector architecture spans heterogeneous backend types (filesystems, object stores, HDF5, NetCDF, IOWarp, external catalogs), each declaring its capabilities so the system adapts its operators to match. An agentic multi-hop loop refines queries across backends based on intermediate results.
\end{enumerate}
