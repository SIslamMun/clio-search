\section{Implementation}

CLIO is implemented as an embedded Python system with no external
service dependencies beyond an optional LLM endpoint. We describe the
software stack and storage schema (Sec.~4.1), connectors for
scientific data formats and external catalogs (Sec.~4.2), and the
experimental platform (Sec.~4.3).

\subsection{Software Stack and Storage}

The system is implemented in Python 3.11+ (13{,}100 lines, 60 modules)
and exposes a CLI (\texttt{clio query}, \texttt{clio index},
\texttt{clio serve}) and an async HTTP API via FastAPI. Dense retrieval
uses sentence-transformers with all-MiniLM-L6-v2 (22.7M parameters,
384-dim, Apache 2.0). Two ANN backends are supported behind a common
\texttt{ANNAdapter} protocol: \texttt{ExactANNAdapter} (16 shards,
exact cosine) and \texttt{HnswANNAdapter} (hnswlib). Write safety on
shared filesystems uses \texttt{fcntl} file locking. OpenTelemetry
instrumentation provides observability with structured trace events
from all retrieval branches.

All persistent state resides in a single DuckDB file
(Table~\ref{tab:schema}). DuckDB was selected for three properties:
embedded operation (no service to deploy on compute nodes), SQL-native
full-text search for BM25 scoring, and single-file compatibility with
POSIX parallel filesystems. The schema comprises eight tables; the two
science-aware tables (\texttt{scientific\_measurements},
\texttt{scientific\_formulas}) store canonicalized SI values and
normalized formula signatures, enabling direct SQL range predicates
that bypass the embedding pipeline.

The agentic layer comprises three modules:
\texttt{CorpusProfile} queries DuckDB aggregate statistics per namespace
(document count, measurement count, metadata density);
\texttt{BranchSelector} decides which retrieval branches to activate;
and \texttt{AgenticRetriever} drives the multi-hop loop with
per-hop token tracking. Connectors opt into profiling via the
\texttt{CorpusProfileCapable} protocol. Additional indexing modules
handle CSV parsing with automatic unit detection
(\texttt{csv\_parser}), data quality flag extraction
(\texttt{quality}), and sampling-based schema recovery for sparse
metadata (\texttt{sample\_schema}). The system is validated by 283
automated tests executed via pytest.

\subsection{Connectors}

Nine connectors implement the \texttt{NamespaceConnector} protocol and
declare capabilities via runtime-checkable protocols:

\textbf{Filesystem and S3} are the reference implementations, supporting
all five capability protocols (lexical, vector, scientific, metadata
filter, corpus profile). Incremental re-indexing uses SHA-256 content
hashes and modification times.

\textbf{HDF5} uses h5py to walk group/dataset hierarchies, extracting
attribute-level units (\texttt{units}, \texttt{long\_name}) and
routing them through SI canonicalization. \textbf{NetCDF} uses xarray
with Climate and Forecast (CF) convention awareness~\cite{cfconventions},
ensuring that a pressure variable in hPa matches the same query as one
in Pa.

\textbf{IOWarp CTE} bridges CLIO to IOWarp's tiered blob storage.
The connector enumerates blobs via \texttt{BlobQuery}, reads content
via \texttt{Tag.GetBlob}, and parses JSON, CSV, or plain text via a
format-detection pipeline. An \texttt{index\_from\_texts} fast path
indexes pre-loaded content at ingest time. The connector implements
all four search capabilities, making IOWarp a first-class CLIO backend.

\textbf{NDP (MCP)} discovers datasets from the National Data Platform
by spawning \texttt{ndp-mcp} as a subprocess and calling
\texttt{search\_datasets} per term via the Model Context Protocol.
It optionally downloads CSV resources and extracts row-level
measurements, enabling CLIO to search live external catalogs (NOAA,
NASA, DOE) with the same science-aware operators used for local data.

\textbf{Qdrant} and \textbf{Neo4j} use in-memory stub implementations
that validate the protocol and capability negotiation logic.
\textbf{KV log store} supports lexical search over key-value and
streaming log data.

\subsection{Experimental Platform}

Single-node experiments run on Linux 6.17 with Python 3.11 and
DuckDB 1.1. Distributed experiments run on NCSA DeltaAI (NVIDIA GH200
Grace Hopper nodes) with 2.5M arXiv abstracts across 4 DuckDB shards;
a lightweight coordinator--worker architecture over TCP/JSON handles
fan-out and result merging without MPI. IOWarp experiments run inside
Docker (\texttt{iowarp/deps-cpu}, Chimaera v1.0.3, 16\,GB shared
memory). The agentic efficiency experiment uses
\texttt{claude-sonnet-4-20250514} via the Claude Agent SDK.

The project uses uv for dependency resolution; \texttt{uv sync}
installs all dependencies from a locked manifest. The full source code,
test suite, and evaluation scripts are open-source.

\textbf{Limitations.} The Qdrant and Neo4j connectors are stubs.
IOWarp integration is validated at 10K blobs; the 10B-blob target
requires production IOWarp infrastructure with NVMe tiers. The
distributed coordinator is an evaluation prototype, not yet part of
the main codebase.
